Q1: Your audio pipeline is currently synchronous per-turn (record → transcribe → LLM → TTS → loop). As you scale toward the menubar app and wake-word activation, should you move to a fully event-driven architecture with an async message bus between pipeline stages?
A) Keep the sequential loop, add async within stages — You already have async for the LLM stream. Keep the linear flow but make each stage internally concurrent (e.g., Whisper model preloading, which you already do). Pros: simple mental model, easy to debug, your 140 tests don't need rewriting. Cons: you can't overlap stages (e.g., start listening while TTS is still finishing the last sentence), limits barge-in sophistication.
B) Full event-driven architecture with asyncio queues between stages — Each stage (VAD, STT, LLM, TTS) runs as an independent async task consuming from input queues and producing to output queues. Pros: true pipeline parallelism, natural fit for barge-in (TTS and VAD run concurrently by design), enables features like "start transcribing chunk N while chunk N-1 is still being spoken." Cons: significantly more complex state management, harder to reason about error propagation, race conditions become a real concern.
C) Actor model using something like aioreactive or a lightweight pub/sub — Each pipeline stage is an actor with its own mailbox. Messages flow between actors. Pros: cleanest separation of concerns, scales to multi-user if you ever need it, actors can be independently tested and restarted. Cons: over-engineered for a single-user CLI tool, adds a dependency, the Python actor ecosystem is immature compared to Erlang/Akka.
Recommendation: B. The asyncio queue approach is the sweet spot. Your barge-in feature already proves you need concurrency between TTS playback and VAD monitoring. Queues between stages formalize what you're already doing informally with speech_started events. It doesn't require a new dependency, it's idiomatic Python asyncio, and it directly enables wake-word and push-to-talk as just another producer on the VAD queue. The actor model is elegant but solves problems you don't have yet.

Q4: Your TTS pipeline buffers at sentence boundaries before speaking. This introduces latency — the user waits for a full sentence before hearing anything. Should you switch to sub-sentence streaming for lower time-to-first-audio?
A) Keep sentence-level buffering — Your current approach produces natural speech with proper prosody. Splitting mid-sentence means TTS engines can't apply proper intonation. Pros: natural-sounding output, simpler implementation, works well with all TTS engines. Cons: if Claude's first sentence is long (e.g., "Let me walk you through the three main things happening in your git status..."), the user waits several seconds before hearing anything.
B) Hybrid: flush on first clause boundary, then sentence boundaries — For the first TTS chunk, split at the first comma, semicolon, or em-dash. After that, revert to sentence boundaries. Pros: dramatically reduces time-to-first-audio (user hears something within 1-2 seconds), subsequent speech still sounds natural. Cons: first clause may sound slightly clipped depending on TTS engine, adds complexity to the buffering logic.
C) Character-level streaming with ElevenLabs' streaming API — ElevenLabs supports websocket streaming where you send text incrementally and get audio back incrementally. Pros: lowest possible latency, the API handles prosody across chunks. Cons: only works with ElevenLabs (not macOS say or OpenAI), significantly more complex audio pipeline, you need to handle audio chunk stitching, and you lose your clean speak_streamed() abstraction.
Recommendation: B. The hybrid approach gives you the best UX trade-off. Users perceive responsiveness from the first audio chunk, and you maintain natural speech quality for the bulk of the response. The implementation is a small change to your sentence boundary regex — add , and ; as flush triggers only for the first buffered chunk, then revert to .!?\n. This works with every TTS engine, preserves your abstraction, and is testable with your existing test infrastructure.

Q3: Your conversation memory stores raw messages but has no summarization. With memory_max_messages=40, long sessions will hit the limit and older context silently drops. How should you handle conversation history that exceeds the window?

Pros: preserves long-term context, user never loses important decisions from earlier in the session, standard pattern in LLM apps. Cons: summarization can lose nuance, adds an LLM call (latency + cost), the summary itself consumes tokens that could be used for actual messages.
B) Importance-weighted retention — Score each message pair by relevance: messages with code references, file paths, or explicit decisions score higher. Cons: fact extraction is an LLM task with accuracy concerns, knowledge store can accumulate stale facts, needs conflict resolution when facts change.
Recommendation: A. Sliding window with summarization is proven and predictable. Pin that summary as a synthetic "system" message at the top of the history. Now you have a summary + 20 recent messages, giving you room to grow again. Run summarization asynchronously after the current turn completes so it never adds user-facing latency.
